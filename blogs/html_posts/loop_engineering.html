Loop Engineering: designing agents that prompt themselves

The job is shifting from typing prompts to building the system that types them — and Claude Code now ships the primitives to do it. Here's how loops, schedules, routines, and monitors actually work, and which one to reach for.

next fire · 09:00

For a few years, working with an AI coding agent meant holding its hand the whole way. You wrote a prompt, read the output, pointed out what was broken, and asked for a revision. You were the orchestrator; the model was a very fast typist. That arrangement is quietly ending.

Loop engineering is the practice of moving from prompting an AI to designing the system that prompts it. Instead of treating code generation as a one-shot query, you structure the work as an iterative, self-correcting cycle: you define a goal, and the agent inspects the codebase, makes a bounded change, runs a verification step like a test suite or linter, reads the result, and decides its next move — repeating until the success criteria are met or a stopping rule fires.

The mental model is less "assistant" and more "control system." You are not writing the answer; you are specifying the objective, the allowed actions, the signal that tells you it's working, and the conditions under which the loop ends. Get those four things right and the agent can run for an hour while you do something else. Get them wrong and you've built a token incinerator.

01 — the shape of the work

The anatomy of a good loop

Unbounded retries are the failure mode everyone hits first. They waste tokens, churn the codebase, and quietly paper over bad assumptions. A loop worth running has four distinct, deliberately designed parts.

  • A clear objective. Exact behavior, not vibes. "Reduce dashboard load time by deferring non-critical charts while keeping the filters working" — something you could verify.
  • Bounded action. The specific, limited changes the agent is permitted to make on each pass.
  • Reliable verification. An observation you can trust: a green test suite, zero lint errors, a successful build.
  • A strict stopping condition. Explicit rules for "done" and for "stuck." For example: stop after five iterations with no measurable progress and hand it back to a human.
02 — patterns worth stealing

Lessons from the Loop Library

The Forward Future Loop Library is a useful catalog of repeatable agent workflows — full operational cycles rather than raw prompts. Three are worth internalizing because they show what a real stopping condition looks like.

  • The Fresh-Clone loop. The agent clones the repo into a throwaway environment and follows the README. When a step fails, it records the gap, fixes the docs, tears the environment down, and starts over. It stops only when a completely clean clone runs the app with zero manual intervention.
  • The Test Stabilizer loop. The agent runs the suite N times to surface flaky tests, then hunts the root cause — shared state, timing — instead of papering over it with retries or sleeps. It stops after N consecutive clean full-suite runs.
  • The 5-Minute Maintainer loop. Wakes on a short cadence, triages open threads, runs tests, enforces review, and merges green CI work without trampling coherent work in progress.

Notice the pattern: each loop ends on a provable condition, not a feeling. That's the whole game.

03 — the toolbox

Claude Code's four native primitives

You no longer need to bolt loop engineering on with bash scripts and cron jobs. Recent versions of Claude Code ship four building blocks that map cleanly onto the practice. They are easy to confuse because the names overlap, so it's worth being precise about each.

/loop — session-scoped polling

The fastest way to run a prompt on repeat while a session stays open. It's a bundled skill (requires Claude Code v2.1.72+). You give it an interval and a prompt; Claude converts the interval to a cron expression, schedules the job, and confirms the cadence and an 8-character job ID.

# interval is a bare token (5m, 30m, 2h) — there is no --interval flag
> /loop 5m check if the deployment finished and tell me what happened

# you can loop another command, too
> /loop 20m /review-pr 1234

# omit the interval and Claude picks one each pass (1 min–1 hr) based on what it sees
> /loop check whether CI passed and address any review comments

Under the hood it uses three tools — CronCreate, CronList, CronDelete — which you can also drive in plain language ("what scheduled tasks do I have?", "cancel the deploy check"). A bare /loop runs a built-in maintenance prompt, or your own .claude/loop.md if one exists. Press Esc to stop a loop while it waits.

Get the details right Two numbers float around the community incorrectly. A session holds up to 50 scheduled tasks at once, and recurring loops expire 7 days after creation (not 3) — the task fires one last time, then deletes itself. If you need it to last longer, recreate it, or graduate to a Routine or a Desktop task.

There's a sibling worth knowing: /goal, which keeps the session working turn after turn until a condition is met, rather than firing on an interval. That's the native version of the "bounded autonomous execution" pattern community wrappers like ralph-claude-code were built to provide — reach for the built-in primitives first.

Desktop scheduled tasks — durable local automation

When you need a recurring task that survives restarts and can touch local files (including uncommitted changes), use the Claude desktop app's scheduled tasks. They run on your machine, persist across restarts, and don't need a session open — but your computer does need to be awake and the app running when the task fires. You create them in the app (its "Routines" sidebar → New → choose Local, as opposed to Remote).

Routines — cloud automation that runs while you sleep

A routine is a saved Claude Code configuration — a prompt, one or more repositories, and a set of connectors — that runs on Anthropic-managed cloud infrastructure. Your laptop can be closed. It's currently in research preview, so limits and the API surface may change.

The real shift isn't capability — Claude Code could already do this work interactively — it's the trigger infrastructure. A routine can fire three ways, and you can combine them on one routine:

  • Scheduled. A recurring cadence (hourly, nightly, weekly) or a one-off future run.
  • API. An HTTP POST to a per-routine endpoint with a bearer token — wire your monitoring tool or CD pipeline to it.
  • GitHub. Repository events like pull_request.opened or a published release.

Each run gets a fresh clone from the default branch and, by default, can only push to claude/-prefixed branches — a sane guardrail so a sloppy prompt never touches main. You create routines at claude.ai/code/routines, in the desktop app (New routine → Remote), or from the CLI with /schedule.

The /schedule footgun In the CLI, /schedule now creates a cloud routine (schedule-trigger only — add API or GitHub triggers later on the web). So "scheduled task" is ambiguous: it can mean a local desktop task, the session-scoped cron behind /loop, or a scheduled routine. When precision matters, name the surface, not just the schedule.

The Monitor tool — event-driven, not a slash command

This is the one most people get wrong: there is no /monitor command you type. The Monitor tool (Claude Code v2.1.98+, shipped April 2026) is something Claude invokes when you ask it to watch something. It runs a small background script and streams each line of stdout back as an event, so Claude reacts the moment something lands — and spends nothing while the stream is quiet.

That flips the economics. Polling with /loop every two minutes over a ten-minute build costs you a handful of full passes whether or not anything changed. A monitor sits silent and wakes once, on the line that matters.

A monitor uses the same permission rules as Bash, stops when you ask Claude to cancel it (or when the session ends), and is never restored on resume. It isn't available on Bedrock, Vertex AI, or Microsoft Foundry.

06 — which one

A decision path

When you're staring at a recurring task and don't know which lever to pull, walk it down from the strongest constraint: does it need to run when you're offline?

Most real setups end up combining them: a routine handles always-on cloud work like PR review and deploy verification, a desktop task runs your local morning checks, /loop babysits a build while you're heads-down, and a monitor watches the dev server so you don't have to. The orchestrator role doesn't disappear — it moves up a level. You stop typing the prompts and start designing the systems that do.

That's loop engineering. The agent is no longer a fast typist. It's a process you've put a control loop around — and your job is to make that loop tight, observable, and bounded.
